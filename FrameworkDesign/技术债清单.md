# 技术债清单

## evidence_log_excerpt 末尾截断对 ASan 不友好

**问题描述**：PoC 阶段 run_verify.yaml 和 verify 阶段 verify_result.yaml 的 log_excerpt 字段，超过 2KB 时只保留尾部。但 ASan 报告关键信息（错误类型行）在开头，截尾会丢失关键诊断信息。

**影响范围**：
- source/app/stages/poc.py（`_build_run_verify_report.evidence_log_excerpt`）
- source/app/stages/verify.py（pre/post log excerpt 字段，目前同款策略）

**建议方案**：实现 `extract_crash_signature(execution_logs) -> str`，按已知 crash 报告类型（ASan / SIGSEGV / abort）做模式提取，失败再退化到截尾。

**优先级**：低。不影响判定正确性，只影响人工排障体验。

## BuildArtifact 缺少显式 project_dir_inside_image 字段

**问题描述**：verify 阶段 `_resolve_project_dir` 用启发式（按 `binary_or_entrypoint` 路径段数推导）猜测容器内 project 目录，对短路径（`/usr/local/bin/lua` 这种）和深路径都可能推错。当前回退到 `${PROJECT_DIR}` 环境变量。

**影响范围**：source/app/stages/verify.py（`_resolve_project_dir`）。当 LLM 重写 Dockerfile 不导出 PROJECT_DIR 且 `binary_or_entrypoint` 路径不规则时，verify 可能 cd 到错误目录。

**建议方案**：在 `BuildArtifact` 加显式 `project_dir_inside_image: str` 字段，由 build 阶段（无论 LLM 规划还是启发式 fallback）直接产出，verify 直接读取，启发式作为兜底。

**优先级**：中。当前默认 Dockerfile 模板都导出 PROJECT_DIR，触发条件是"LLM dockerfile_override 且不导出 PROJECT_DIR"——窄但真实。

## environment_variables 透传 docker run，无 allowlist

**问题描述**：PoC 阶段 plan 里的 `environment_variables` 在 PoC 和 verify 阶段都被原样应用到 docker run。如果 LLM 规划器塞入 `LD_PRELOAD` / `PATH` / `SUDO_ASKPASS` 等敏感变量，可能改变容器行为绕过判定。

**影响范围**：
- source/app/stages/poc.py（`_execute_poc_plan` docker run 调用）
- source/app/stages/verify.py（`_run_one_pass`）

**建议方案**：在 PoC 阶段加 env 名 allowlist（基于已知合理变量集，如 `ASAN_OPTIONS` / `UBSAN_OPTIONS` / `LUA_PATH` / `PYTHONPATH` 等）。verify 不需要单独加，PoC 过滤后 verify 直接复用就是干净的。

**优先级**：低。当前 PoC 阶段是受控环境（PoC 文件、trigger 命令都来自同一个 plan），未观测到实际滥用；建立 allowlist 需要预先调研合理 env 集合，工作量不小。
